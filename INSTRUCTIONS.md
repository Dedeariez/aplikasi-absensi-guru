### Bagian 1: Setup Backend Supabase

Backend aplikasi ini menggunakan Supabase untuk otentikasi, database, dan fitur realtime.

#### Langkah 1: Buat Project Baru di Supabase
1.  Login ke akun Supabase Anda.
2.  Klik **"New project"**.
3.  Pilih organisasi, berikan nama project (misal: `absensi-darul-inayah`), dan buat password database yang kuat.
4.  Pilih region yang paling dekat dengan Anda (misal: Southeast Asia/Singapore).
5.  Klik **"Create new project"** dan tunggu beberapa menit hingga project Anda siap.

#### Langkah 2: Dapatkan Kunci API
1.  Setelah project siap, navigasi ke **Project Settings** (ikon roda gigi).
2.  Pilih menu **API**.
3.  Di halaman ini, Anda akan menemukan **Project URL** dan **Project API Keys**.
4.  Anda akan membutuhkan `URL` dan `anon` `public` key untuk langkah selanjutnya.

#### Langkah 3: Setup Skema Database dengan SQL
1.  Navigasi ke **SQL Editor** di menu sebelah kiri.
2.  Klik **"+ New query"**.
3.  Salin dan tempelkan seluruh kode SQL di bawah ini ke dalam editor. Kode ini akan membuat tabel yang dibutuhkan (`profiles`, `students`, `attendance`, `audit_log`) dan mengatur RLS (Row Level Security) untuk keamanan data.
4.  Klik **"RUN"** untuk mengeksekusi script.

```sql
-- 1. EXTENSIONS (jika belum aktif)
create extension if not exists "uuid-ossp" with schema "extensions";

-- 2. ENUMS (Custom Types)
create type public.user_role as enum ('teacher', 'parent');
create type public.student_gender as enum ('L', 'P');
create type public.attendance_status as enum ('Hadir', 'Sakit', 'Izin', 'Alpa', 'Tidur');

-- Helper function to check for super admin
create or replace function public.is_super_admin()
returns boolean as $$
begin
  return (select auth.jwt() ->> 'email') = 'ayahnieda@gmail.com';
end;
$$ language plpgsql security definer;

-- 3. TABLE: PROFILES
-- Menyimpan data publik pengguna, terhubung dengan tabel auth.users
create table public.profiles (
  id uuid not null references auth.users on delete cascade,
  full_name text,
  role user_role,
  email text,
  primary key (id)
);
-- Atur agar RLS aktif untuk tabel ini
alter table public.profiles enable row level security;
-- Policy: Pengguna bisa melihat semua profil, mengubah profil sendiri, dan super admin bisa menghapus.
create policy "Public profiles are viewable by everyone." on public.profiles for select using (true);
create policy "Users can insert their own profile." on public.profiles for insert with check (auth.uid() = id);
create policy "Users can update own profile." on public.profiles for update using (auth.uid() = id);
create policy "Super Admins can delete users." on public.profiles for delete using (is_super_admin());


-- 4. TABLE: STUDENTS
-- Menyimpan data siswa
create table public.students (
  id uuid not null primary key default uuid_generate_v4(),
  full_name text not null,
  grade integer not null,
  class_letter char(1) not null,
  gender student_gender not null,
  nisn text null, -- Kolom baru untuk NISN (opsional)
  parent_id uuid references public.profiles on delete set null,
  created_at timestamp with time zone default timezone('utc'::text, now()) not null
);
-- Atur RLS
alter table public.students enable row level security;
-- Policy: Guru bisa melihat dan mengubah semua data siswa. Orang tua hanya bisa melihat data anaknya. Super admin bisa hapus.
create policy "Teachers can manage students." on public.students for all using ((select role from public.profiles where id = auth.uid()) = 'teacher') with check ((select role from public.profiles where id = auth.uid()) = 'teacher');
create policy "Parents can view their own children." on public.students for select using ((select role from public.profiles where id = auth.uid()) = 'parent' and parent_id = auth.uid());
create policy "Super Admins can delete students." on public.students for delete using (is_super_admin());


-- 5. TABLE: ATTENDANCE
-- Menyimpan catatan absensi
create table public.attendance (
  id bigint generated by default as identity primary key,
  student_id uuid not null references public.students on delete cascade,
  date date not null,
  lesson_hour integer not null,
  status attendance_status not null,
  taken_by_teacher_id uuid references public.profiles on delete set null,
  created_at timestamp with time zone default timezone('utc'::text, now()) not null,
  unique (student_id, date, lesson_hour)
);
-- Atur RLS
alter table public.attendance enable row level security;
-- Policy: Guru bisa mengelola absensi. Orang tua bisa melihat absensi anaknya. Super admin bisa hapus.
create policy "Teachers can manage attendance." on public.attendance for all using ((select role from public.profiles where id = auth.uid()) = 'teacher') with check ((select role from public.profiles where id = auth.uid()) = 'teacher');
create policy "Parents can view their own child's attendance." on public.attendance for select using ( (select role from public.profiles where id = auth.uid()) = 'parent' and student_id in (select id from public.students where parent_id = auth.uid()) );
create policy "Super Admins can delete attendance." on public.attendance for delete using (is_super_admin());


-- 6. TABLE: AUDIT LOG
-- Menyimpan log aktivitas penting
create table public.audit_log (
  id bigint generated by default as identity primary key,
  created_at timestamp with time zone default timezone('utc'::text, now()) not null,
  user_email text,
  action text not null
);
-- Atur RLS
alter table public.audit_log enable row level security;
-- Policy: Semua pengguna terautentikasi bisa melihat log, tapi hanya guru/admin yang bisa menambah log baru. Super admin bisa hapus.
create policy "Authenticated users can view audit logs." on public.audit_log for select using (auth.role() = 'authenticated');
create policy "Allow insert for teachers and admins" on public.audit_log for insert with check (((select role from public.profiles where id = auth.uid()) = 'teacher' OR is_super_admin()));
create policy "Super Admins can delete logs." on public.audit_log for delete using (is_super_admin());


-- 7. FUNCTION & TRIGGER
-- Fungsi ini akan menyalin data dari auth.users ke public.profiles saat user baru mendaftar
create or replace function public.handle_new_user()
returns trigger as $$
begin
  insert into public.profiles (id, full_name, role, email)
  values (new.id, new.raw_user_meta_data->>'full_name', (new.raw_user_meta_data->>'role')::public.user_role, new.email);
  return new;
end;
$$ language plpgsql security definer;

-- Trigger yang akan menjalankan fungsi di atas setiap kali ada user baru
create trigger on_auth_user_created
  after insert on auth.users
  for each row execute procedure public.handle_new_user();

```

#### Langkah 4: Aktifkan Realtime
Fitur log aktivitas membutuhkan Supabase Realtime.
1. Navigasi ke **Database** -> **Replication**.
2. Anda akan melihat tabel-tabel Anda. Cari tabel `audit_log`.
3. Klik tombol di kolom "Source" di sebelah `audit_log` untuk mengaktifkan replikasi.
4. Di dialog yang muncul, centang kotak `INSERT`.
5. Klik **"Save"**.

Backend Anda sekarang siap!

### Bagian 2: Setup Project di Lokal

#### Langkah 1: Clone Repository
Buka terminal/CMD dan jalankan perintah:
```bash
git clone <URL_REPOSITORY_ANDA>
cd <NAMA_FOLDER_PROJECT>
```

#### Langkah 2: Install Dependencies
Aplikasi ini membutuhkan beberapa pustaka pihak ketiga. Install semuanya dengan satu perintah:
```bash
npm install
```

#### Langkah 3: Buat File Environment Variable
Untuk alasan keamanan, kunci API Supabase harus disimpan dalam file environment.
1. Buat file baru di folder root project dengan nama `.env`.
2. Salin dan tempelkan konten di bawah ini ke dalam file `.env` tersebut.
3. Ganti `URL_SUPABASE_ANDA` dan `KUNCI_ANON_SUPABASE_ANDA` dengan kredensial dari project Supabase Anda.

```
VITE_SUPABASE_URL="URL_SUPABASE_ANDA"
VITE_SUPABASE_ANON_KEY="KUNCI_ANON_SUPABASE_ANDA"
```
Aplikasi secara otomatis akan menggunakan kunci dari file ini saat berjalan di lokal.

#### Langkah 4: Jalankan Aplikasi
```bash
npm run dev
```
Aplikasi Anda sekarang berjalan di `http://localhost:5173` (atau port lain yang tersedia).

### Bagian 3: Deployment ke Vercel

#### Langkah 1: Push Kode ke GitHub
Pastikan semua perubahan Anda sudah di-commit dan di-push ke repository GitHub Anda.

#### Langkah 2: Hubungkan Project di Vercel
1.  Login ke Vercel dan klik **"Add New... -> Project"**.
2.  Pilih repository GitHub yang ingin Anda deploy.
3.  Vercel akan otomatis mendeteksi bahwa ini adalah project Vite. Pengaturan default biasanya sudah benar.
4.  **PENTING**: Buka bagian **"Environment Variables"**.
5.  Tambahkan dua variabel (ini adalah cara aman untuk menyimpan kunci API di production):
    *   **Name**: `VITE_SUPABASE_URL`, **Value**: (tempelkan URL Supabase Anda)
    *   **Name**: `VITE_SUPABASE_ANON_KEY`, **Value**: (tempelkan Kunci Anon Supabase Anda)
6.  Klik **"Deploy"**. Vercel akan membangun dan mendeploy aplikasi Anda. Setelah selesai, Anda akan mendapatkan URL publik.

### Bagian 4: Cara Memperbarui Aplikasi yang Sudah Ada

Jika Anda ingin memperbarui aplikasi yang sudah berjalan di Vercel dan Supabase.

#### Langkah 1: Update Kode
1.  Selalu mulai dengan mengambil perubahan terbaru dari server: `git pull origin main`.
2.  Buat branch baru untuk fitur atau perbaikan Anda: `git checkout -b nama-fitur-baru`.
3.  Lakukan perubahan kode yang Anda inginkan.
4.  Commit dan push branch Anda ke GitHub: `git push -u origin nama-fitur-baru`.
5.  Buka GitHub dan buat **Pull Request (PR)** dari branch Anda ke branch `main`.
6.  Setelah direview, **Merge** PR tersebut.

Vercel akan secara otomatis mendeteksi perubahan pada branch `main` dan memulai proses deployment baru dengan kode terbaru.

#### Langkah 2: Update Database (Jika Ada Perubahan Skema)
**JANGAN** mengubah skema database produksi langsung dari SQL Editor Supabase jika sudah ada data penting. Gunakan **Migrations**:
1.  Install Supabase CLI: `npm install -g supabase`.
2.  Hubungkan CLI ke project Anda: `supabase login` lalu `supabase link --project-ref <ID_PROJECT_ANDA>`.
3.  Buat file migrasi baru untuk perubahan Anda: `supabase migration new nama_perubahan_skema`.
4.  Tulis perintah SQL untuk perubahan (misal: `ALTER TABLE students ADD COLUMN nisn TEXT;`) di dalam file migrasi yang baru dibuat.
5.  Terapkan migrasi ke database Supabase: `supabase db push`.

Ini adalah cara yang aman untuk mengelola perubahan skema database tanpa merusak data yang sudah ada.

### Bagian 5: Penyelesaian Masalah (Troubleshooting)

**Error: "Kredensial Supabase tidak ditemukan..."**

Jika Anda melihat error ini saat membuka aplikasi di browser, itu berarti aplikasi tidak dapat menemukan kunci API Supabase Anda. Ini bisa terjadi karena beberapa alasan:

1.  **Server `npm run dev` tidak berjalan**: Metode yang direkomendasikan untuk menjalankan aplikasi secara lokal adalah dengan perintah `npm run dev` di terminal. Perintah ini akan memulai server pengembangan Vite yang secara otomatis memuat variabel dari file `.env`. **Jangan membuka file `index.html` secara langsung di browser**, karena metode tersebut tidak dapat memuat kredensial Anda.

2.  **File `.env` salah**: Periksa kembali file `.env` Anda. Pastikan namanya tepat (`.env`, bukan `.env.txt`), berada di folder root project (di level yang sama dengan `index.html`), dan berisi `VITE_SUPABASE_URL` dan `VITE_SUPABASE_ANON_KEY` dengan nilai yang benar.

**Solusi Alternatif (Tidak Direkomendasikan)**:
Jika karena suatu alasan Anda tidak bisa menggunakan `npm run dev`, Anda dapat mengedit file `services/supabaseService.ts` secara langsung dan mengganti placeholder `URL_SUPABASE_ANDA` dan `KUNCI_ANON_SUPABASE_ANDA` dengan kredensial Anda. Namun, ini tidak disarankan karena Anda bisa secara tidak sengaja membagikan kunci rahasia Anda.
